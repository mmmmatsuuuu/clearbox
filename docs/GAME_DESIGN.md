# ゲーム設計ドキュメント

## コンセプト

「授業で習った 16 進数が、なんの役に立つのかわからない」という感覚を解消するゲーム。  
プレイヤーはゲームを攻略するために自然とセーブデータを調べ、バイナリ表現を実感する。

## 対象プレイヤー

- 16 進数・2 進数の基礎を授業で習っている中学〜高校生
- Chromebook でプレイ
- プレイ時間の目安：全部わかっていれば 1 時間程度（詰まった場合の上限なし）

## ゲーム構成

### 舞台設定

魔王が頂上に住む「塔」を舞台にした 2D ドット絵 RPG。  
プレイヤーの目的は塔を登り、最上階の魔王を倒すこと。

### フロア構成

| 階 | 内容 |
|---|---|
| -1F | 裏ボス（魔王クリア後に解放。実装は後フェーズ） |
| 1F | チュートリアル（NPC からゲームの仕組みと護符を受け取る） |
| 2F | ボス 1 |
| 3F | ボス 2 |
| 4F | ボス 3 |
| 5F | ボス 4 |
| 最上階 | 魔王 |

### 学習の段階設計

| 階 | 担当する概念 | 仕掛け | チート発覚の条件 |
|---|---|---|---|
| 2F | uint8 / 16 進数の基本 | HP を増やして生き残る | HP > 255 で発覚（「駆け出しの勇者が 255 超はあり得ない」） |
| 3F | 符号付き整数（int8） | 通れない座標（X = -1 = `0xFF`）の技を取得する | なし（罠として機能） |
| 4F | uint16 リトルエンディアン | HP の 2 バイト目を使って 255 超にしないと勝てない | なし（2F で「255 が上限」と思い込ませた上での発見） |
| 5F | NPC 交流コード | 1F NPC のコードを予測して NPC 交流スロットに設定 | なし |
| 最上階 | XOR チェックサム | 護符の値が合わないと初期ステータスにリセットされて戦闘 | 護符の不整合 → 初期ステータスにリセット（ゲームオーバーではない） |
| -1F | float32（後フェーズ） | 魔王から入手した技の威力を float32 として操作 | — |

### 2F → 4F の意図的な罠

2F で「HP が 255 を超えるとチートがバレる」という体験をさせることで、プレイヤーは HP の上限を 255 と思い込む。  
4F でどう頑張っても 255 HP では負ける状況を作り、「HP に 2 バイト目があるのでは？」という発見を促す。

### 3F の座標改ざん

通常は壁に囲まれてどう歩いても到達できない場所に技がある。  
座標は int8 で保存されており、X = -1 は `0xFF` として表現される。  
2F で「`0xFF` = 255（uint8）」と学んでいるため、「`0xFF` は int8 では -1 になる」という気づきが必要になる。

### 5F の NPC コード予測

セーブデータには倒した敵を連番コード（0x01, 0x02, ...）で記録するスロットがある。  
このパターンを読み取ることで、NPC コード体系の規則性に気づける。  
1F の NPC のコードを NPC 交流スロットに設定すると、その NPC から技を受け取った扱いになり、5F ボスに必要な技が手に入る。

### 魔王と護符

1F の NPC から「誠実の護符」を受け取る。  
護符の値はセーブデータの全フィールドの XOR チェックサム。  
魔王だけがこの意味を知っており、護符の値が合わない場合は初期ステータスにリセットして戦闘を開始する。  
ゲームオーバーにはならないため、プレイヤーは弱い状態で魔王と戦うことになる。

## スキルシステム

- プレイヤーは最大 4 つのスキルを習得できる
- スキルは各フロアのダンジョンや NPC から入手する
- 入手時にスキルコード（1 バイト）が告知される（例：「ファイアのコード: `0xA3`」）
- セーブデータのスキルスロット（0x06〜0x09）に正しいコードをセットすることでスキルを装備する
- スキルの威力がそのままボスへのダメージになる（攻撃力というパラメータは存在しない）

## チート発覚の演出方針

- メタ的な発言はせず、没入感を維持する
- 発覚タイミングは NPC に話しかけたとき・ボス戦開始時
- 「ゲームオーバー」ではなく「自然な結果」として表現する

| 発覚条件 | セリフ例 | 結果 |
|---|---|---|
| HP > 255（2F ボス） | 「そなたの生命力…あり得ぬ数値だ」 | ゲームオーバー |
| 護符の不整合（魔王） | 「誠実の護符が濁っておる。貴様不正をしておるな！元のステータスに戻してくれる！」 | 初期ステータスにリセットして戦闘開始 |

## マップシステム

- 各フロアのマップは最大 **16×16 タイル**
- 画面には勇者を中心に **5×5 タイル**（canvas: 320×320px、タイルサイズ: 64px）のみ表示
- カメラが勇者を追従し、移動で全体を把握する探索型レイアウト

## ゲームオーバー

- 勇者 HP が 0 になった場合、またはチート発覚でゲームオーバーになった場合
- `SaveManager.reset()` で全ステートを初期化しタイトルへ戻る
- ロードで再開できるため、セーブの重要性を体験させる

## 2F 設計

### ボス：キングスライム

| 項目 | 値 |
|---|---|
| HP | 40 |
| 攻撃力 | 3 |
| ヒール | HP ≤ 20 で 1 回のみ全回復（HP 40 に戻る） |
| 登場 | 勇者と衝突でバトル開始（セリフなし） |
| チート発覚 | バトル開始時 HP > 255 → 「そなたの生命力…あり得ぬ数値だ。駆け出しの勇者がそれほどの力を持てるはずがない。リセットだ〜」→ ゲームオーバー |

### 斬撃ダメージ改定

初期スキル「斬撃」の威力を 5 → **10** に変更。

### 2F NPC

| 配置 | セリフ |
|---|---|
| 入口付近 | 「ここのボスは粘り強いらしいぞ！頑張れよ！」 |
| 中盤 | 「おまえそんな HP で大丈夫か？」 |
| 奥の部屋（側道経由） | 「ボスに勝てない？もうセーブデータをいじるしか無いよな？どこをいじればいいかって？それは自分で考えろ！」 |

### 勝利条件の目安

- 最低 HP 16 以上で理論上撃破可能（余裕を持つなら 30 前後）
- HP ≤ 255 の範囲でセーブデータの 1 バイト目を書き換えることで解決できる

## UI / UX

- ドット絵スタイル（旧ドラクエ風）
- 操作：キーボード（矢印 + Z / X）または画面タップ（Chromebook 対応）
- セーブ：ボタンクリックで `save_data.txt` をダウンロード
- ロード：ファイルアップロードボタンで `.txt` を選択

### ステータス画面

```
┌───────────────────────┐
│ HP  : 10              │
│ 護符: ✦ 光っている    │  ← チェックサムが正常
│ スキル: なし          │
└───────────────────────┘
```

ステータス画面を開くたびにチェックサムを計算し、護符の値と照合する。

| 状態 | 表示 | 意味 |
|---|---|---|
| チェックサム一致 | ✦ 光っている | セーブデータの整合性が保たれている |
| チェックサム不一致 | ◆ くすんでいる | どこかのフィールドが護符と合っていない |

護符の値をステータス画面で直接表示しないことで、「護符とは何か」という謎を魔王戦まで持続させる。

## セーブ・ロードの UX フロー

```
[セーブ]
プレイヤーが「セーブ」を選択
→ save_data.txt をブラウザからダウンロード
→ "セーブしました" メッセージ

[ロード]
タイトル画面の「ロード」ボタン
→ ファイル選択ダイアログ
→ ファイルを読み込み・検証（マジックバイト・バイト数）
→ ゲーム再開
```

## ヒントの設計方針

- NPC・看板などで「セーブデータのここを調べると…」レベルのヒントを出す
- 答えをそのまま教えるのではなく「どのオフセットを見ればいいか」程度にとどめる
- ヒントの量・質はテストプレイで調整する
